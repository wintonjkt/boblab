Security Inline

Shift-Left

The Challenge

Security reviews typically happen days after code is written. Finding vulnerabilities late means expensive rework and delayed releases.

Watch Bob Catch Security Issues

Real-Time Security Detection

Issue Detected: Hardcoded JWT Secret
Initial Generation 
const token = jwt.sign(payload, 'secret-key');
→
Bob Self-Corrected 
const token = jwt.sign(payload, process.env.JWT_SECRET);

Bob's explanation: "Hardcoded secrets are a security risk. Using environment variables keeps secrets out of source code."

Issue Detected: Token Storage in localStorage
Vulnerable Code 
localStorage.setItem('token', token);
→
Bob Fixed 
// Set httpOnly cookie via server response
res.cookie('token', token, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict'
});

Bob's explanation: "localStorage is vulnerable to XSS attacks. httpOnly cookies prevent JavaScript access to tokens."

Security Issues Bob Caught During Generation

  • Hardcoded JWT secret → Environment variable
  • CSRF protection disabled → Re-enabled with proper config
  • Tokens in localStorage → httpOnly secure cookies
  • Missing session timeout → 15-minute enforcement added

Business Value

Traditional Approach:

Security review 3-5 days later → find more issues → rework cycle → delays

With Bob:

Security scanning during generation → zero debt accumulation → no rework

"Shift-left security - vulnerabilities are caught before code is written, eliminating the security-review-rework cycle."

Deployment Planning

Zero-Downtime

The Challenge

Deploying across multiple systems requires careful orchestration to avoid downtime. Rollback procedures must be planned before deployment.

Generate Deployment Runbook

1

Request Deployment Plan

Prompt to Bob 
Generate a zero-downtime deployment runbook for this migration.
Include deployment order, health checks, rollback procedures,
and cutover strategy.
2

Deployment Runbook Generated

Bob creates a comprehensive runbook including:

Phase 1: Pre-Deployment
  • Backup current auth tokens from Redis
  • Take database snapshots of user_service DB
  • Verify all environment variables are set
  • Configure feature flags for gradual rollout
Phase 2: Deploy auth-service (Blue-Green)
  • Deploy new version to green environment
  • Run health checks: GET /health, GET /api/mfa/status
  • Test MFA enrollment with test user
  • If healthy, switch traffic (5% → 50% → 100%)
  • If unhealthy, rollback: switch traffic back to blue
Phase 3: Deploy user-service (Rolling Update)
  • Update pods one at a time (maxUnavailable: 1)
  • Health check: GET /actuator/health after each pod
  • Verify OAuth2 integration with auth-service
  • Rollback command: kubectl rollout undo deployment/user-service
Phase 4: Deploy frontend (Canary)
  • Deploy to canary servers (10% of traffic)
  • Monitor error rates, auth failures, session timeouts
  • Gradual increase: 10% → 25% → 50% → 100%
  • Rollback: redirect canary traffic back to stable
Rollback Triggers
  • Error rate > 0.1% for 5 minutes
  • MFA verification failure rate > 5%
  • Auth response time P95 > 500ms
  • Any critical security logs

Business Value

Traditional Approach:

2-3 days - write runbook manually, test rollback procedures, document steps

With Bob:

45 seconds - comprehensive runbook with health checks and rollback commands

"Zero-downtime deployments - Bob plans deployment order, health checks, and rollback procedures before any code is deployed."
