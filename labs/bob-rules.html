What Are Custom Rules?

Custom rules extend Bob's default behavior by defining preferences, constraints, and guidelines that shape how Bob interacts with your code and responds to your requests.

Understanding Custom Rules

Common Uses for Custom Rules

📝 Coding Style

Define indentation, naming conventions, and formatting preferences

📚 Documentation

Specify documentation formats and standards for your project

🧪 Testing

Set testing methodologies and coverage requirements

🔄 Workflows

Define project workflows and development processes

👥 Team Conventions

Enforce team-specific conventions and best practices

💡 Key Insight: Custom rules act as a persistent context that Bob considers with every interaction, ensuring consistency across your development workflow.

Adding Custom Rules for a Mode

You can add custom rules directly through Bob's interface for any mode. These rules will apply whenever you're working in that specific mode.

Step-by-Step Guide

How to Add Mode-Specific Rules

  1. Click the three dots in the top-right corner of the Bob sidebar
  2. Click "Modes" from the dropdown menu
  3. Select the mode you want to edit rules for
  4. Click "Edit" to open the mode configuration
  5. Scroll to "Mode-specific custom instructions (optional)"
  6. Type your rules in the text area
  7. Click "Save" to apply your changes

Example: Code Mode Rules

Mode-Specific Instructions 
When in Code mode:
- Always show a summary of changes before applying
- Run tests after making changes
- Update related documentation
- Check for unused imports and remove them
- Ensure code formatting matches project style
📝 Note: Mode-specific rules only apply when you're actively using that mode, allowing you to have different behaviors for different tasks.

Rule Scopes: Global vs Workspace

Custom rules can be defined at two different scopes, each serving different purposes in your development workflow.

Understanding Rule Scopes

🌍 Global Rules

Apply automatically across all projects

  • Personal coding preferences
  • Organization-wide standards
  • Language-specific conventions
  • General best practices
Use for: Standards that should apply everywhere you work

📁 Workspace Rules

Apply only within the current project

  • Project-specific requirements
  • Team conventions for this project
  • Technology stack preferences
  • Custom workflows
Use for: Project-specific requirements and team standards
💡 Best Practice: Use global rules for personal and organization-wide standards, and workspace rules for project-specific requirements. Workspace rules can override global rules when needed.

Global Custom Instructions

Global instructions apply to all modes and interactions with Bob. Use them to enforce coding standards, documentation requirements, and general best practices across your entire project.

Global Instructions Example

Coding Standards Example

Global Instructions 
Always:
- Use TypeScript with strict mode enabled
- Include comprehensive error handling with try-catch blocks
- Write unit tests for all new functions
- Follow ESLint rules defined in .eslintrc
- Add JSDoc comments for all public APIs
- Use async/await instead of promises
- Prefer functional programming patterns
- Keep functions under 50 lines of code

Documentation Requirements

Documentation Rules 
Documentation Standards:
- Every module must have a README.md
- All functions must have JSDoc with @param and @returns
- Include usage examples in documentation
- Document all environment variables
- Maintain CHANGELOG.md for version history
- Add inline comments for complex logic

Security & Performance

Security Rules 
Security & Performance:
- Never commit secrets or API keys
- Use environment variables for configuration
- Validate all user inputs
- Sanitize data before database operations
- Implement rate limiting for APIs
- Use prepared statements for SQL queries
- Optimize database queries with indexes
- Implement caching where appropriate
📝 Note: Global instructions are always active and apply to every interaction with Bob, regardless of the current mode.

Mode-Specific Custom Instructions

Mode-specific instructions apply only when Bob is in a particular mode (Code, Plan, Ask, Debug, or custom modes). This allows you to tailor Bob's behavior for specific tasks.

Mode-Specific Instructions

Code Mode Instructions

Code Mode Rules 
When in Code mode:
- Always show a summary of changes before applying
- Maintain backward compatibility
- Include migration notes if breaking changes are needed
- Run tests after making changes
- Update related documentation
- Check for unused imports and remove them
- Ensure code formatting matches project style

Debug Mode Instructions

Debug Mode Rules 
When in Debug mode:
- Start with the most likely causes
- Check logs and error messages first
- Suggest adding debug logging if needed
- Provide step-by-step debugging approach
- Recommend using debugger breakpoints
- Check for common pitfalls (null checks, async issues)
- Suggest relevant debugging tools

Plan Mode Instructions

Plan Mode Rules 
When in Plan mode:
- Break down tasks into small, actionable steps
- Estimate time for each step
- Identify dependencies between tasks
- Consider edge cases and error scenarios
- Suggest testing strategies
- Highlight potential risks
- Recommend architectural patterns when relevant
🎯 Strategy: Use mode-specific instructions to create specialized workflows that activate only when needed, keeping Bob focused on the task at hand.

Best Practices

Follow these guidelines to create effective custom instructions that improve your development workflow.

✅ Be Specific

Use concrete, actionable instructions rather than vague guidelines.

❌ Vague: "Write good code"
✅ Specific: "Use TypeScript strict mode, add JSDoc comments, and write unit tests"

📋 Prioritize Instructions

Put the most important rules first. Bob pays more attention to earlier instructions.

Prioritized Instructions 
Priority Order:
1. Security requirements (highest priority)
2. Code quality standards
3. Documentation requirements
4. Performance considerations
5. Style preferences (lowest priority)

🔄 Keep Instructions Updated

Review and update your instructions as your project evolves and team practices change.

  • Review quarterly or when major changes occur
  • Remove outdated or conflicting rules
  • Add new rules based on lessons learned
  • Get team feedback on effectiveness

🎯 Use Mode-Specific Wisely

Don't duplicate global instructions in mode-specific ones. Use mode-specific for truly mode-dependent rules.

❌ Redundant: Adding "Use TypeScript" to every mode when it's already global
✅ Mode-Specific: "In Code mode: Show diff preview before applying changes"

📏 Keep It Concise

Aim for clarity over comprehensiveness. Too many instructions can be overwhelming.

  • Focus on 5-10 key rules per category
  • Use bullet points for readability
  • Avoid redundancy
  • Group related rules together

🧪 Test Your Instructions

Verify that Bob follows your instructions by testing with real tasks.

  • Try common development tasks
  • Check if Bob follows the rules
  • Refine instructions based on results
  • Share effective patterns with team

Advanced Techniques

Advanced Instruction Patterns

Conditional Instructions

Create instructions that apply based on context or file types.

Conditional Rules 
Context-Aware Rules:
- For .test.ts files: Focus on test coverage and edge cases
- For .api.ts files: Emphasize input validation and error handling
- For .component.tsx files: Prioritize accessibility and performance
- For .config.ts files: Ensure type safety and documentation
- For migration files: Include rollback procedures

Workflow Instructions

Define multi-step workflows that Bob should follow.

Workflow Pattern 
Feature Development Workflow:
1. Analyze requirements and ask clarifying questions
2. Create a plan with tasks and estimates
3. Implement core functionality with tests
4. Add error handling and edge cases
5. Update documentation
6. Run tests and fix any issues
7. Create a summary of changes
