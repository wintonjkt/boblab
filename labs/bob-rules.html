What Are Custom Rules?

Custom rules extend Bob's default behavior by defining preferences, constraints, and guidelines that shape how Bob interacts with your code and responds to your requests.

Understanding Custom Rules

Common Uses for Custom Rules

📝 Coding Style

Define indentation, naming conventions, and formatting preferences

📚 Documentation

Specify documentation formats and standards for your project

🧪 Testing

Set testing methodologies and coverage requirements

🔄 Workflows

Define project workflows and development processes

👥 Team Conventions

Enforce team-specific conventions and best practices

💡 Key Insight: Custom rules act as a persistent context that Bob considers with every interaction, ensuring consistency across your development workflow.

Adding Custom Rules for a Mode

You can add custom rules directly through Bob's interface for any mode. These rules will apply whenever you're working in that specific mode.

Step-by-Step Guide

How to Add Mode-Specific Rules

  1. Click the three dots in the top-right corner of the Bob sidebar
  2. Click "Modes" from the dropdown menu
  3. Select the mode you want to edit rules for
  4. Click "Edit" to open the mode configuration
  5. Scroll to "Mode-specific custom instructions (optional)"
  6. Type your rules in the text area
  7. Click "Save" to apply your changes

Example: Code Mode Rules

Mode-Specific Instructions 
When in Code mode:
- Always show a summary of changes before applying
- Run tests after making changes
- Update related documentation
- Check for unused imports and remove them
- Ensure code formatting matches project style
📝 Note: Mode-specific rules only apply when you're actively using that mode, allowing you to have different behaviors for different tasks.

Rule Scopes: Global vs Workspace

Custom rules can be defined at two different scopes, each serving different purposes in your development workflow.

Understanding Rule Scopes

🌍 Global Rules

Apply automatically across all projects

  • Personal coding preferences
  • Organization-wide standards
  • Language-specific conventions
  • General best practices
Use for: Standards that should apply everywhere you work

📁 Workspace Rules

Apply only within the current project

  • Project-specific requirements
  • Team conventions for this project
  • Technology stack preferences
  • Custom workflows
Use for: Project-specific requirements and team standards
💡 Best Practice: Use global rules for personal and organization-wide standards, and workspace rules for project-specific requirements. Workspace rules can override global rules when needed.

Configuration Methods

Bob supports multiple ways to configure custom rules, from simple file-based configuration to organized directory structures.

File-Based Configuration

Simple File-Based Rules

Create files in your workspace root to define rules:

File Configuration 
.bobrules              # General workspace rules
.bobrules-code         # Rules for Code mode
.bobrules-plan         # Rules for Plan mode
.bobrules-{modeSlug}   # Rules for any custom mode

Example: Creating a .bobrules file

bash 
echo "Use 4 spaces for indentation" > .bobrules
Directory-Based Configuration

Workspace Directory Structure

Workspace Structure 
.bob/
├── rules/              # General rules for all modes
│   └── coding-style.md
└── rules-code/         # Code mode specific rules
    └── typescript.md

Global Directory Structure

Global Structure 
# Linux/macOS
~/.bob/rules/

# Windows
%USERPROFILE%\.bob\rules\

Creating Workspace Rules

bash 
mkdir -p .bob/rules
echo "# Coding Standards" > .bob/rules/coding-style.md
echo "- Use TypeScript strict mode" >> .bob/rules/coding-style.md

Creating Global Rules

bash 
mkdir -p ~/.bob/rules
echo "# Personal Preferences" > ~/.bob/rules/preferences.md
echo "- Always explain reasoning before code" >> ~/.bob/rules/preferences.md
💡 Tip: Directory-based configuration is more flexible and allows you to organize rules into multiple files for better maintainability.

Rule Priority

Understanding how Bob combines and prioritizes rules is essential for effective configuration.

Rule Loading Order

How Rules Are Combined

Bob loads rules in the following order:

Priority Order 
1. Global rules (~/.bob/rules/)
   ↓
2. Workspace rules (.bob/rules/)

Within each level:
- Mode-specific rules load before general rules
- Workspace rules can override global rules

Example Scenario

Rule Combination Example 
# Global rule says: "Use 2 spaces for indentation"
~/.bob/rules/style.md

# Workspace rule says: "Use 4 spaces for indentation"
.bob/rules/style.md

# Result: Bob uses 4 spaces (workspace overrides global)
⚡ Key Point: Workspace rules take precedence over global rules, allowing you to customize behavior for specific projects while maintaining organization-wide defaults.

Writing Effective Rules

Well-written rules are specific, actionable, and clearly structured. Follow these guidelines to create rules that Bob can effectively follow.

Rule Writing Best Practices

Be Specific and Actionable

❌ Vague: "Format code nicely"
✅ Specific: "Use 4 spaces for indentation in JavaScript files"
❌ Vague: "Write good tests"
✅ Specific: "Write unit tests for all public functions using Jest"

Use Clear Structure

Well-Structured Rules 
# Code Style
- Use camelCase for variables
- Use PascalCase for classes
- Use UPPER_CASE for constants

# Testing
- Write unit tests for all public functions
- Use Jest for testing framework
- Aim for 80% code coverage

# Documentation
- Add JSDoc comments for public APIs
- Include usage examples in documentation
- Document all environment variables

Examples of Effective Rules

Effective Rule Examples 
✅ "Always use spaces for indentation, with a width of 4 spaces"
✅ "Use camelCase for variable names"
✅ "Write unit tests for all new functions"
✅ "Explain your reasoning before providing code"
✅ "When adding features to websites, ensure they are responsive and accessible"
✅ "Use async/await instead of promise chains"
✅ "Include error handling in all API calls"
📝 Remember: The more specific and actionable your rules are, the better Bob can follow them consistently across your project.

Advanced Configuration

Take your custom rules to the next level with advanced configuration options including mode-specific directories, AGENTS.md files, and understanding file behavior.

Mode-Specific Rules Directories

Organizing Rules by Mode

Create separate directories for different modes to keep rules organized:

Mode-Specific Directories 
.bob/
├── rules/          # General rules for all modes
├── rules-code/     # Code mode only
├── rules-plan/     # Plan mode only
├── rules-debug/    # Debug mode only
└── rules-{mode}/   # Any custom mode

Example: Creating Mode-Specific Rules

bash 
mkdir -p .bob/rules-code
echo "# Code Mode Rules" > .bob/rules-code/standards.md
echo "- Always run tests after changes" >> .bob/rules-code/standards.md
echo "- Update documentation for API changes" >> .bob/rules-code/standards.md
Using AGENTS.md Files

AGENTS.md for Team Standardization

The AGENTS.md file provides a standardized way to define rules that are automatically loaded by Bob:

  • Location: Place in workspace root
  • Automatic loading: Loaded by default without configuration
  • Version control: Commit with your project for team consistency
  • Priority: Loaded after mode-specific rules but before general workspace rules

Example AGENTS.md

markdown 
# Team Development Standards

## Code Quality
- Use TypeScript strict mode
- Maintain 80% test coverage
- Follow ESLint configuration

## Documentation
- Update README for new features
- Add JSDoc for public APIs
- Document breaking changes in CHANGELOG

## Git Workflow
- Use conventional commits
- Create feature branches from main
- Require PR reviews before merge

Disabling AGENTS.md

If needed, you can disable AGENTS.md loading in VS Code settings:

json 
{
  "bob-cline.useAgentRules": false
}
File Behavior

Understanding How Bob Processes Rule Files

📂 Recursive Reading

Reads all files in rules directories, including subdirectories

🔤 Alphabetical Order

Files are processed alphabetically by filename

🗑️ Automatic Filtering

Excludes cache files (.DS_Store, *.bak, *.cache, *.log, *.tmp, Thumbs.db)

🔗 Symbolic Links

Supported with maximum depth of 5 levels

📄 Empty Files

Silently skipped without errors

Organizing Rules with Prefixes

Use numeric prefixes to control loading order:

File Organization 
.bob/rules/
├── 01-security.md      # Loaded first (highest priority)
├── 02-code-style.md    # Loaded second
├── 03-testing.md       # Loaded third
└── 99-preferences.md   # Loaded last (lowest priority)
💡 Pro Tip: Use subdirectories and numeric prefixes to organize complex rule sets while maintaining clear priority and loading order.

Organization-Wide Standards

Distributing Global Rules

Share organization-wide standards by distributing global rules to all team members:

Option 1: Shared Rules Repository

bash 
# Create a shared rules repository
git clone https://github.com/your-org/bob-rules.git
cd bob-rules

# Team members clone to their global rules directory
git clone https://github.com/your-org/bob-rules.git ~/.bob/rules

# Update periodically
cd ~/.bob/rules
git pull

Option 2: Setup Script

bash 
#!/bin/bash
# setup-bob-rules.sh

mkdir -p ~/.bob/rules
curl -o ~/.bob/rules/org-standards.md \
  https://raw.githubusercontent.com/your-org/bob-rules/main/standards.md

echo "Organization rules installed successfully!"

Hybrid Setup

Best Practices

Follow these guidelines to create effective custom instructions that improve your development workflow.

✅ Be Specific

Use concrete, actionable instructions rather than vague guidelines.

❌ Vague: "Write good code"
✅ Specific: "Use TypeScript strict mode, add JSDoc comments, and write unit tests"

📋 Prioritize Instructions

Put the most important rules first. Bob pays more attention to earlier instructions.

Prioritized Instructions 
Priority Order:
1. Security requirements (highest priority)
2. Code quality standards
3. Documentation requirements
4. Performance considerations
5. Style preferences (lowest priority)

🔄 Keep Instructions Updated

Review and update your instructions as your project evolves and team practices change.

  • Review quarterly or when major changes occur
  • Remove outdated or conflicting rules
  • Add new rules based on lessons learned
  • Get team feedback on effectiveness

🎯 Use Mode-Specific Wisely

Don't duplicate global instructions in mode-specific ones. Use mode-specific for truly mode-dependent rules.

❌ Redundant: Adding "Use TypeScript" to every mode when it's already global
✅ Mode-Specific: "In Code mode: Show diff preview before applying changes"

📏 Keep It Concise

Aim for clarity over comprehensiveness. Too many instructions can be overwhelming.

  • Focus on 5-10 key rules per category
  • Use bullet points for readability
  • Avoid redundancy
  • Group related rules together

🧪 Test Your Instructions

Verify that Bob follows your instructions by testing with real tasks.

  • Try common development tasks
  • Check if Bob follows the rules
  • Refine instructions based on results
  • Share effective patterns with team
