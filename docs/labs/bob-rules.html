What Are Custom Rules?

5 minutes

Custom rules are instructions that guide Bob's behavior when working with your code. They help ensure consistency and adherence to your team's standards.

Common Uses

  • Coding standards: Enforce specific formatting, naming conventions, or architectural patterns
  • Framework preferences: Specify which libraries or frameworks to use
  • Documentation requirements: Ensure code includes proper comments and documentation
  • Testing practices: Define testing approaches and coverage requirements
  • Security guidelines: Enforce security best practices and prevent common vulnerabilities
2

Adding Custom Rules for a Mode

10 minutes

You can add custom rules through the Bob interface:

Step-by-Step Process

1

Open Bob Settings

Click the settings icon in Bob's interface

2

Navigate to Custom Instructions

Select "Custom Instructions" from the settings menu

3

Choose Scope

Select either "Global" or "Workspace" scope for your rules

4

Select Mode

Choose the mode you want to add rules for (e.g., Code, Architect, etc.)

5

Write Your Rules

Enter your custom instructions in the text area

6

Save

Click "Save" to apply your rules

7

Test

Verify that Bob follows your rules in the selected mode

Rule Scopes

5 minutes

Custom rules can be applied at different scopes:

Global Rules

Global rules apply to all workspaces and projects. Use them for:

  • Personal preferences that apply everywhere
  • Organization-wide standards
  • General coding practices

Workspace Rules

Workspace rules apply only to the current workspace. Use them for:

  • Project-specific requirements
  • Team conventions
  • Technology stack preferences

When to Use Each

  • Global: For rules that should apply across all your work
  • Workspace: For project-specific or team-specific rules that might differ between projects

Configuration Methods

10 minutes

Besides the UI, you can configure rules using files:

File-Based Configuration

Create rule files in your project root:

plaintext 
.bobrules              # General rules for all modes
.bobrules-code         # Rules specific to Code mode
.bobrules-{modeSlug}   # Rules for any specific mode

Directory-Based Configuration

Organize rules in directories:

plaintext 
.bob/rules/            # Workspace rules directory
~/.bob/rules/          # Global rules directory

Creating Rule Files

Example of creating a workspace rule file:

bash 
# Create a general rules file
echo "Always use TypeScript for new files" > .bobrules

# Create a Code mode specific rules file
echo "Follow the project's ESLint configuration" > .bobrules-code
5

Rule Priority

5 minutes

When multiple rules exist, Bob applies them in this order:

  1. Global rules are applied first
  2. Workspace rules are applied second, potentially overriding global rules

Within each scope, mode-specific rules take precedence over general rules:

  • .bobrules-code overrides .bobrules when in Code mode
  • .bob/rules/code.md overrides .bob/rules/general.md when in Code mode

💡 Key Point

Workspace rules can override global rules, allowing project-specific customization while maintaining organization-wide defaults.

Writing Effective Rules

10 minutes

Be Specific and Actionable

Write clear, concrete instructions rather than vague guidelines.

❌ Bad Example

plaintext 
Write good code

✅ Good Example

plaintext 
Use camelCase for variable names and PascalCase for class names. 
Add JSDoc comments for all public functions.

Use Clear Structure

Organize rules with headers and bullet points for readability:

markdown 
# Naming Conventions
- Use camelCase for variables and functions
- Use PascalCase for classes and components
- Use UPPER_SNAKE_CASE for constants

# Documentation
- Add JSDoc comments for all public APIs
- Include examples in documentation
- Document all parameters and return values

Examples of Effective Rules

markdown 
# Example 1: Framework Preference
Always use React hooks instead of class components. 
Prefer functional components with useState and useEffect.

# Example 2: Testing Requirements
Write unit tests for all business logic functions.
Aim for at least 80% code coverage.
Use Jest for testing and React Testing Library for component tests.

# Example 3: Error Handling
Always use try-catch blocks for async operations.
Log errors with context information.
Never expose sensitive information in error messages.

# Example 4: Code Organization
Keep components under 200 lines.
Extract complex logic into custom hooks.
Use barrel exports (index.ts) for cleaner imports.

# Example 5: Security
Never commit API keys or secrets.
Use environment variables for configuration.
Sanitize all user inputs before processing.

Advanced Configuration

10 minutes

Mode-Specific Rules

Create separate rule files for different modes:

plaintext 
.bob/rules/code.md       # Rules for Code mode
.bob/rules/architect.md  # Rules for Architect mode
.bob/rules/ask.md        # Rules for Ask mode
Directory Scope Description
~/.bob/rules/ Global Rules that apply to all workspaces
.bob/rules/ Workspace Rules specific to current workspace

Advanced Configuration

For complex rule sets, consider organizing them into themed collections and using project-level configuration.

File Behavior

  • Bob automatically detects and loads rule files when you open a workspace
  • Changes to rule files take effect immediately
  • You can have multiple rule files that are combined
  • Mode-specific rules override general rules
  • Workspace rules override global rules

Team Standardization

5 minutes

Project-Level Standards

Commit rule files to your repository:

bash 
git add .bobrules .bob/
git commit -m "Add Bob custom rules for team"
git push

This ensures all team members use the same rules when working on the project.

Organization-Wide Standards

For organization-wide standards:

  1. Create a shared repository for Bob rules
  2. Document the rules and their purpose
  3. Have team members copy rules to their ~/.bob/rules/ directory
  4. Update rules centrally and communicate changes

Hybrid Approach

Combine global and workspace rules:

  • Global rules: Organization-wide standards that apply everywhere
  • Workspace rules: Project-specific requirements that may differ

This allows flexibility while maintaining consistency where it matters.
