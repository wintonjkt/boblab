What is Spec-Driven Development

Foundation

Definition

Spec-driven development is an approach where you start with clear, comprehensive specifications before implementation. These specifications serve as a contract between stakeholders, designers, and developers, ensuring everyone has a shared understanding of what needs to be built.

Key Benefits

📋 Clear Requirements

Everyone knows exactly what needs to be built, reducing back-and-forth and rework.

🎯 Reduced Ambiguity

Precise specifications eliminate misunderstandings and misinterpretations.

📊 Better Planning

Accurate effort estimation and sprint planning based on detailed specs.

✅ Easier Testing

Acceptance criteria in specs translate directly to test cases.

🤝 Improved Collaboration

Shared understanding across product, design, and engineering teams.

📚 Living Documentation

Specs serve as up-to-date documentation for the entire system.

When to Use Spec-Driven Development

  • Complex Features: When building features with multiple components and interactions
  • Team Collaboration: When multiple developers work on related functionality
  • API Development: When defining contracts between services or systems
  • Regulated Industries: When compliance and audit trails are required
  • Long-term Projects: When maintainability and knowledge transfer are critical

The Spec-Driven Workflow

Process

Six-Step Process

Follow this structured workflow to implement spec-driven development effectively.

Step 1: Write the Specification

Document requirements, features, user stories, and acceptance criteria before writing any code.

What to Include:
  • Feature overview and purpose
  • User stories and use cases
  • Functional requirements
  • Technical requirements
  • Acceptance criteria
  • Edge cases and error handling

Step 2: Review and Refine

Collaborate with stakeholders, designers, and team members to validate and improve the specification.

Review Checklist:
  • Are requirements clear and unambiguous?
  • Are all edge cases covered?
  • Do acceptance criteria align with user needs?
  • Are technical constraints documented?
  • Is the scope well-defined?

Step 3: Plan Implementation

Break down the specification into actionable tasks, estimate effort, and create a development plan.

Planning Activities:
  • Identify components and modules
  • Define dependencies and order
  • Estimate development time
  • Assign tasks to team members
  • Create milestones and checkpoints

Step 4: Implement

Build the feature according to the specification, referencing it throughout development.

Implementation Best Practices:
  • Reference spec sections in code comments
  • Implement one requirement at a time
  • Write tests based on acceptance criteria
  • Ask questions when spec is unclear
  • Document deviations from spec

Step 5: Validate

Ensure the implementation matches the specification and meets all acceptance criteria.

Validation Methods:
  • Run tests against acceptance criteria
  • Perform manual testing of user flows
  • Review code against spec requirements
  • Conduct stakeholder demos
  • Verify edge cases and error handling

Step 6: Update

Keep the specification in sync with any changes made during or after implementation.

Maintenance Tasks:
  • Document approved changes to requirements
  • Update acceptance criteria if modified
  • Add lessons learned and gotchas
  • Archive outdated sections
  • Keep spec as single source of truth

Creating Effective Specifications

Best Practices

Writing Specs That Work

Learn how to write specifications that are clear, comprehensive, and actionable.

📝

Use Clear Language

Write in simple, unambiguous terms that everyone can understand.

Good: "The system shall display an error message when the user enters an invalid email address."

Bad: "Handle email validation appropriately."
👤

Include User Stories

Frame requirements from the user's perspective to maintain focus on value.

Format: "As a [user type], I want to [action] so that [benefit]."

Example: "As a customer, I want to reset my password so that I can regain access to my account."
✅

Define Acceptance Criteria

Specify measurable conditions that must be met for the feature to be considered complete.

Example Criteria:
  • User can submit form with valid data
  • Error messages appear for invalid fields
  • Success message displays on submission
  • Form data is saved to database
⚙️

Specify Technical Requirements

Document technical constraints, dependencies, and implementation details.

Include:
  • Performance requirements (response time, throughput)
  • Security requirements (authentication, authorization)
  • Integration points (APIs, databases)
  • Technology stack and frameworks
🔍

Include Examples

Provide concrete examples and edge cases to clarify requirements.

Example: "Valid email formats: user@example.com, user.name@example.co.uk
Invalid formats: user@, @example.com, user@example"
📐

Document Assumptions

Clearly state assumptions and constraints that affect the implementation.

Example Assumptions:
  • Users have JavaScript enabled
  • Maximum file upload size is 10MB
  • System supports modern browsers only
  • API rate limit is 100 requests/minute
Example Specification Structure 
# User Authentication Feature

## Overview
Implement secure user authentication with email/password login.

## User Stories
- As a user, I want to log in with my email and password
- As a user, I want to reset my password if I forget it
- As a user, I want to stay logged in across sessions

## Functional Requirements
1. Login form with email and password fields
2. "Remember me" checkbox for persistent sessions
3. "Forgot password" link to password reset flow
4. Input validation with clear error messages
5. Redirect to dashboard after successful login

## Technical Requirements
- Use JWT tokens for authentication
- Hash passwords with bcrypt (cost factor: 12)
- Session timeout: 24 hours (7 days if "remember me")
- Rate limiting: 5 failed attempts per 15 minutes
- HTTPS required for all authentication endpoints

## Acceptance Criteria
- [ ] User can log in with valid credentials
- [ ] Invalid credentials show appropriate error
- [ ] "Remember me" extends session to 7 days
- [ ] Password reset email sent within 1 minute
- [ ] Account locked after 5 failed attempts
- [ ] All authentication uses HTTPS

## Edge Cases
- Empty email or password fields
- Email not found in system
- Incorrect password
- Account already locked
- Expired password reset token
- Concurrent login attempts

## API Endpoints
POST /api/auth/login
POST /api/auth/logout
POST /api/auth/forgot-password
POST /api/auth/reset-password

## Dependencies
- User database table
- Email service for password reset
- Redis for rate limiting

## Assumptions
- Users have valid email addresses
- Email service is reliable
- Users can receive and click email links

Practical Example: Task Management API

Hands-On

Complete Walkthrough

Follow this end-to-end example of building a task management API using spec-driven development with Bob.

Phase 1: Create the Specification

Start by creating a comprehensive specification for the task management API.

Prompt to Bob (Plan Mode) 
💬 "Create a specification for a RESTful task management API with the following features:
- Create, read, update, delete tasks
- Task properties: title, description, status, priority, due date
- Filter tasks by status and priority
- Mark tasks as complete
- Assign tasks to users
Include user stories, acceptance criteria, API endpoints, and error handling."
Generated Specification (specs/task-api.md) 
# Task Management API Specification

## Overview
RESTful API for managing tasks with CRUD operations, filtering, and user assignment.

## User Stories
- As a user, I want to create tasks with title, description, and due date
- As a user, I want to view all my tasks
- As a user, I want to filter tasks by status and priority
- As a user, I want to mark tasks as complete
- As a user, I want to assign tasks to team members

## API Endpoints

### Create Task
POST /api/tasks
Request Body: { title, description, status, priority, dueDate, assignedTo }
Response: 201 Created with task object

### Get All Tasks
GET /api/tasks?status=pending&priority=high
Response: 200 OK with array of tasks

### Get Task by ID
GET /api/tasks/:id
Response: 200 OK with task object or 404 Not Found

### Update Task
PUT /api/tasks/:id
Request Body: { title, description, status, priority, dueDate, assignedTo }
Response: 200 OK with updated task

### Delete Task
DELETE /api/tasks/:id
Response: 204 No Content

### Mark Complete
PATCH /api/tasks/:id/complete
Response: 200 OK with updated task

## Data Model
Task {
  id: string (UUID)
  title: string (required, max 200 chars)
  description: string (optional, max 2000 chars)
  status: enum ['pending', 'in-progress', 'completed']
  priority: enum ['low', 'medium', 'high']
  dueDate: ISO 8601 date string
  assignedTo: string (user ID)
  createdAt: ISO 8601 timestamp
  updatedAt: ISO 8601 timestamp
}

## Acceptance Criteria
- [ ] Can create task with valid data
- [ ] Cannot create task without title
- [ ] Can retrieve all tasks
- [ ] Can filter by status and priority
- [ ] Can update task fields
- [ ] Can delete task
- [ ] Can mark task as complete
- [ ] Returns 404 for non-existent task
- [ ] Returns 400 for invalid data

## Error Responses
400 Bad Request: Invalid input data
404 Not Found: Task not found
500 Internal Server Error: Server error

Phase 2: Review and Refine

Use Bob to review the specification and identify improvements.

Prompt to Bob (Ask Mode) 
💬 "Review @specs/task-api.md and suggest:
1. Missing edge cases
2. Additional validation rules
3. Security considerations
4. Performance optimizations"

Bob's Suggestions Might Include:

  • Edge Cases: Duplicate titles, past due dates, invalid user IDs
  • Validation: Title length limits, date format validation, enum value checks
  • Security: Authentication required, authorization for task access, rate limiting
  • Performance: Pagination for task lists, database indexing, caching strategy

Phase 3: Plan Implementation

Break down the specification into implementation tasks.

Prompt to Bob (Plan Mode) 
💬 "Based on @specs/task-api.md, create an implementation plan with:
1. Database schema
2. API route structure
3. Controller functions
4. Validation middleware
5. Test cases
6. Implementation order"
Implementation Plan 
Implementation Order:

1. Database Setup
   - Create tasks table with schema
   - Add indexes for status, priority, assignedTo
   - Set up migrations

2. Data Models
   - Create Task model with validation
   - Define enums for status and priority
   - Add timestamp handling

3. Validation Middleware
   - Title validation (required, max length)
   - Status/priority enum validation
   - Date format validation
   - User ID validation

4. Controllers
   - createTask: Handle POST /api/tasks
   - getTasks: Handle GET /api/tasks with filters
   - getTaskById: Handle GET /api/tasks/:id
   - updateTask: Handle PUT /api/tasks/:id
   - deleteTask: Handle DELETE /api/tasks/:id
   - markComplete: Handle PATCH /api/tasks/:id/complete

5. Routes
   - Set up Express routes
   - Apply validation middleware
   - Add error handling

6. Tests
   - Unit tests for each controller
   - Integration tests for API endpoints
   - Test all acceptance criteria
   - Test edge cases and error scenarios

Phase 4-6: Implement, Validate, Update

Continue with implementation, validation, and spec updates using the prompts shown in the "Using Bob" section above.

Spec File Organization

Structure

Organizing Your Specifications

A well-organized spec directory makes it easy to find, maintain, and reference specifications.

Recommended Directory Structure 
project/
├── specs/
│   ├── README.md                    # Overview of all specs
│   ├── features/                    # Feature specifications
│   │   ├── authentication.md
│   │   ├── user-management.md
│   │   ├── task-management.md
│   │   ├── notifications.md
│   │   └── reporting.md
│   ├── technical/                   # Technical specifications
│   │   ├── architecture.md
│   │   ├── database-schema.md
│   │   ├── api-design.md
│   │   ├── security.md
│   │   └── performance.md
│   ├── testing/                     # Test specifications
│   │   ├── test-strategy.md
│   │   ├── test-scenarios.md
│   │   └── acceptance-tests.md
│   └── archive/                     # Deprecated specs
│       └── old-features/
├── src/                             # Source code
├── tests/                           # Test files
└── docs/                            # Additional documentation

Naming Conventions

  • Use kebab-case for file names
  • Be descriptive: user-authentication.md not auth.md
  • Group related specs in subdirectories
  • Use consistent prefixes for related specs

README.md Structure

  • List all specifications with brief descriptions
  • Indicate spec status (draft, approved, implemented)
  • Link to related specs and dependencies
  • Include last updated date

Version Control

  • Keep specs in Git alongside code
  • Use meaningful commit messages for spec changes
  • Tag spec versions for major releases
  • Review spec changes in pull requests

Cross-Referencing

  • Link related specs using relative paths
  • Reference spec sections in code comments
  • Maintain a dependency map between specs
  • Use consistent section IDs for linking

Common Pitfalls to Avoid

Watch Out

❌ Vague Specifications

Problem: Writing specs that are too general or ambiguous.

Solution: Use specific language, concrete examples, and measurable criteria.

❌ Over-Specification

Problem: Specifying implementation details too rigidly, limiting flexibility.

Solution: Focus on what needs to be achieved, not how to achieve it.

❌ Stale Specs

Problem: Not updating specs when code changes, leading to drift.

Solution: Make spec updates part of your development workflow.

❌ One-Time Documents

Problem: Treating specs as write-once documents that are never revisited.

Solution: Maintain specs as living documents throughout the project lifecycle.

❌ Skipping Reviews

Problem: Not getting stakeholder buy-in before implementation.

Solution: Always review and approve specs with stakeholders first.

❌ Missing Acceptance Criteria

Problem: Not defining clear success criteria for features.

Solution: Always include measurable acceptance criteria in every spec.

❌ Ignoring Edge Cases

Problem: Focusing only on happy paths and missing error scenarios.

Solution: Explicitly document edge cases and error handling.

❌ Isolated Specs

Problem: Creating specs in isolation without team collaboration.

Solution: Involve developers, designers, and stakeholders in spec creation.
