Configure Documentation MCP Server

What You'll Learn

Set up the remote MCP server that gives Bob access to watsonx Orchestrate documentation. This eliminates the need to manually search docs or reference specific pages.

MCP Server Configuration

1

Open MCP Settings

In Bob, navigate to:

  • Settings → MCP Servers → Add Server
2

Configure Server Details

Add the watsonx Orchestrate documentation MCP server:

MCP Configuration 
{
  "name": "wxo-docs",
  "command": "uvx",
  "args": ["mcp-proxy"],
  "transport": "streamablehttp",
  "url": "https://developer.watson-orchestrate.ibm.com/mcp",
  "allowed_tools": ["SearchIbmWatsonxOrchestrateAdk"]
}
3

Understanding the Configuration

  • command: Uses uvx to run the MCP proxy
  • transport: Streamable HTTP for remote server access
  • url: Official watsonx Orchestrate documentation endpoint
  • allowed_tools: Restricts to documentation search tool only

Configure CLI MCP Server

What You'll Learn

Set up the local MCP server that enables Bob to invoke watsonx Orchestrate CLI commands. This allows Bob to list agents, export configurations, and deploy agents directly.

CLI MCP Configuration

1

Install watsonx Orchestrate Package

First, ensure the watsonx Orchestrate package is available:

Terminal 
# Verify uvx can access the package
uvx --version

# The ibm-watsonx-orchestrate package will be loaded by the MCP server
2

Add CLI MCP Server

In Bob's MCP settings, add a new server:

MCP Configuration 
{
  "name": "wxo-cli",
  "command": "uvx",
  "args": [
    "--with",
    "ibm-watsonx-orchestrate==2.1.0",
    "ibm-watsonx-orchestrate"
  ],
  "env": {
    "WXO_MCP_WORKING_DIRECTORY": "/path/to/your/wxo-projects"
  },
  "allowed_tools": [
    "list_agents",
    "export_agent",
    "list_tools",
    "list_toolkits",
    "create_agent",
    "deploy_agent"
  ]
}

Important: Replace /path/to/your/wxo-projects with your actual project directory path.

3

Understanding CLI Tools

The CLI MCP server provides these tools:

  • list_agents: View all agents in your workspace
  • export_agent: Export agent configuration
  • list_tools: List available tools for agents
  • list_toolkits: View available toolkits
  • create_agent: Create a new agent
  • deploy_agent: Deploy agent to watsonx Orchestrate
4

Test CLI Access

Bob should use the list_agents CLI tool and display your agents.

Generated Files 
# weather_agent.py
from ibm_watsonx_orchestrate import Agent, Tool
import requests
import os

class WeatherAgent(Agent):
    """Agent that retrieves weather information for cities"""
    
    def __init__(self):
        super().__init__(
            name="Weather Agent",
            description="Retrieves current weather information for any city"
        )
        self.api_key = os.getenv('OPENWEATHER_API_KEY')
        self.base_url = "https://api.openweathermap.org/data/2.5/weather"
    
    @Tool(
        name="get_weather",
        description="Get current weather for a city"
    )
    def get_weather(self, city: str) -> dict:
        """
        Retrieve weather information for a specified city.
        
        Args:
            city: Name of the city
            
        Returns:
            Dictionary with weather information
        """
        try:
            params = {
                'q': city,
                'appid': self.api_key,
                'units': 'metric'
            }
            
            response = requests.get(self.base_url, params=params)
            response.raise_for_status()
            
            data = response.json()
            
            return {
                'city': data['name'],
                'temperature': data['main']['temp'],
                'conditions': data['weather'][0]['description'],
                'humidity': data['main']['humidity'],
                'status': 'success'
            }
            
        except requests.exceptions.RequestException as e:
            return {
                'status': 'error',
                'message': f'Failed to retrieve weather: {str(e)}'
            }
        except KeyError as e:
            return {
                'status': 'error',
                'message': f'Invalid response format: {str(e)}'
            }

# Initialize the agent
agent = WeatherAgent()

Best Practices

Maximize Your Success

Follow these best practices to get the most out of Bob and watsonx Orchestrate development.

Development Best Practices

1

Start with Clear Requirements

Provide Bob with specific, actionable requirements:

  • What the agent should do
  • What inputs it needs
  • What outputs it should provide
  • Any specific APIs or tools to use
  • Error handling requirements
2

Let Bob Search Documentation First

Always allow Bob to search the watsonx Orchestrate documentation before generating code. This ensures:

  • Correct ADK patterns are used
  • Best practices are followed
  • Latest features are leveraged
3

Iterate Incrementally

Build agents in stages:

  1. Start with basic functionality
  2. Test and verify
  3. Add features incrementally
  4. Refine based on feedback
4

Exclude ibm-watsonx-orchestrate from requirements.txt

The MCP server handles this dependency. Including it in requirements.txt can cause conflicts.

5

Use Environment Variables for Secrets

Never hardcode API keys or credentials. Use environment variables:

Python Example 
import os

api_key = os.getenv('API_KEY')
if not api_key:
    raise ValueError("API_KEY environment variable not set")
6

Test Locally Before Deploying

Always test agents in your local environment before deploying to watsonx Orchestrate:

Test Prompt 
Create a test script for the Weather Agent and run it locally
7

Document Your Agents

Ask Bob to create comprehensive documentation:

  • Agent purpose and capabilities
  • Required environment variables
  • Usage examples
  • Error handling behavior

MCP Configuration Tips

1

Keep MCP Servers Updated

Regularly check for updates to the watsonx Orchestrate MCP servers to access new features and improvements.

2

Use Specific Tool Allowlists

Only enable the MCP tools you need for security and clarity:

Good Practice 
"allowed_tools": [
  "list_agents",
  "create_agent",
  "deploy_agent"
]
3

Set Correct Working Directory

Ensure the WXO_MCP_WORKING_DIRECTORY environment variable points to your actual project directory.

Common Pitfalls to Avoid

  • Including ibm-watsonx-orchestrate in requirements.txt - This causes dependency conflicts
  • Hardcoding credentials - Always use environment variables
  • Skipping documentation search - Let Bob search docs for best practices
  • Deploying without testing - Always test locally first
  • Vague prompts - Be specific about requirements and constraints
