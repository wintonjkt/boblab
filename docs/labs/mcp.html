Introduction to MCP in Bob

Model Context Protocol (MCP) extends Bob's capabilities by connecting to external tools and services. This guide covers everything you need to know about configuring, managing, and using MCP servers with Bob.

External Integrations

Connect Bob to databases, APIs, file systems, and custom services

Global and project-level settings for team collaboration

Granular permissions and approval workflows

Estimated Time: 20-30 minutes to master MCP configuration and usage

Configuration Levels

Bob supports two levels of MCP configuration, allowing you to manage servers globally or per-project.

Global Configuration

  • Location: mcp_settings.json
  • Scope: Applies across all workspaces
  • Use Case: Personal tools and utilities you use everywhere
  • Example: Database clients, file system tools, personal APIs

Project Configuration

  • Location: .Bob/mcp.json in project root
  • Scope: Specific to the current project
  • Use Case: Project-specific integrations shared with team
  • Example: Project APIs, custom build tools, team services

Configuration Precedence

Important: Project-level configurations take precedence over global settings. If a server is defined in both locations, the project configuration will be used.

This allows teams to share project-specific MCP configurations via version control while maintaining personal global settings.

Editing Configuration Files

There are two convenient ways to edit your MCP configuration files in Bob.

Method 1: Bob Pane Icon 
1. Click the icon in Bob pane top navigation
2. Select "Edit Global MCP" or "Edit Project MCP"
3. Configuration file opens in editor
Method 2: Settings Menu 
1. Open Settings menu
2. Navigate to MCP tab
3. Click "Edit Global Configuration" or "Edit Project Configuration"
4. Configuration file opens in editor

Configuration File Format

MCP configuration files use JSON format with the following structure:

mcp_settings.json / .Bob/mcp.json 
{
  "mcpServers": {
    "server1": {
      "command": "python",
      "args": ["/path/to/server.py"],
      "env": {
        "API_KEY": "your_api_key"
      },
      "alwaysAllow": ["tool1", "tool2"],
      "disabled": false
    }
  }
}

Configuration Parameters

  • command: The executable to run (required)
  • args: Array of command-line arguments (optional)
  • env: Environment variables for the server (optional)
  • alwaysAllow: Array of tool names to auto-approve (optional)
  • disabled: Set to true to disable the server (optional)

Understanding Transport Types

MCP supports two transport types for communication between Bob and MCP servers.

STDIO Transport (Local Servers)

STDIO transport is used for local MCP servers running on your machine.

Benefits

  • Lower latency
  • Better security (no network exposure)
  • Simpler setup
  • Direct process communication

Configuration Parameters

  • command (required): Executable to run
  • args (optional): Command arguments
  • cwd (optional): Working directory
  • env (optional): Environment variables
  • alwaysAllow (optional): Auto-approve tools
  • disabled (optional): Disable server
STDIO Transport Example 
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/allowed/files"],
      "env": {
        "NODE_ENV": "production"
      },
      "alwaysAllow": ["read_file", "list_directory"],
      "disabled": false
    },
    "database": {
      "command": "python",
      "args": ["-m", "mcp_server_postgres"],
      "cwd": "/path/to/server",
      "env": {
        "DB_CONNECTION_STRING": "postgresql://localhost/mydb"
      }
    }
  }
}

SSE Transport (Remote Servers)

SSE (Server-Sent Events) transport is used for remote MCP servers accessible over HTTP.

Benefits

  • Can be hosted on different machines
  • Supports multiple clients
  • Centralized deployment
  • Easier scaling

Configuration Parameters

  • url (required): Full URL endpoint
  • headers (optional): Custom HTTP headers
  • alwaysAllow (optional): Auto-approve tools
  • disabled (optional): Disable server
SSE Transport Example 
{
  "mcpServers": {
    "remote-api": {
      "url": "https://api.example.com/mcp",
      "headers": {
        "Authorization": "Bearer your_token_here",
        "X-Custom-Header": "value"
      },
      "alwaysAllow": ["search", "fetch_data"],
      "disabled": false
    }
  }
}

Enabling or Disabling MCP Servers

Control whether Bob uses MCP servers and whether it can create new ones.

Disabling All MCP Servers

When you disable MCP servers globally, Bob removes MCP-related logic from its system prompt, reducing token usage.

Effects of Disabling MCP Servers:

  • Bob cannot connect to any MCP servers
  • All MCP tools become unavailable
  • Reduces system prompt token usage
  • Useful when MCP functionality is not needed
Steps to Disable MCP Servers 
1. Open Settings menu
2. Navigate to MCP tab
3. Toggle "Use MCP Servers" to OFF
4. MCP functionality is now disabled

Disabling MCP Server Creation

You can disable Bob's ability to create new MCP servers while keeping existing servers functional.

Effects of Disabling Server Creation:

  • Removes server creation instructions from system prompt
  • Reduces token usage
  • Bob retains context for operating existing servers
  • Useful when you only need to use pre-configured servers
Steps to Disable Server Creation 
1. Open Settings menu
2. Navigate to MCP tab
3. Toggle "Enable MCP Server Creation" to OFF
4. Bob can no longer create new servers

Managing Individual MCP Servers

Bob provides a comprehensive interface for managing each MCP server.

Accessing Server Management

Steps to Manage Servers 
1. Open Settings menu
2. Navigate to MCP tab
3. Locate the server you want to manage
4. Use the available controls

Available Actions

Delete Server

Permanently remove a server from your configuration.

  • Click the delete icon next to the server
  • Confirm the deletion
  • Server is removed from configuration

Restart Server

Restart a server to apply changes or fix issues.

  • Click the restart button
  • Server process is terminated and restarted
  • Useful after configuration changes

Enable/Disable Server

Temporarily disable a server without deleting it.

  • Toggle the enable/disable switch
  • Disabled servers are not loaded
  • Easy to re-enable when needed

Network Timeout

Configure how long Bob waits for server responses.

  • Range: 30 seconds to 5 minutes
  • Default: 1 minute
  • Adjust based on server performance

Server Status Indicators

  • 🟢 Connected: Server is running and responsive
  • 🟡 Starting: Server is initializing
  • 🔴 Error: Server encountered an error
  • ⚫ Disabled: Server is disabled

Auto-approve Tools

Configure which MCP tools can be used without manual approval for each invocation.

How Auto-approve Works

  • By default, Bob asks for approval before using any MCP tool
  • Auto-approve can be enabled per-tool, not per-server
  • Useful for frequently used, safe operations
  • Reduces interruptions in your workflow

Enabling Auto-approve

Follow these steps to enable auto-approve for specific tools:

Auto-approve Configuration Steps 
Step 1: Enable Global Auto-approve
  1. Open Settings menu
  2. Navigate to auto-approving actions section
  3. Enable "Use MCP servers" option

Step 2: Configure Per-Tool Auto-approve
  1. Go to Settings → MCP tab
  2. Locate the MCP server
  3. Find the specific tool you want to auto-approve
  4. Check "Always allow" next to the tool name

Note: The global setting takes precedence. If global auto-approve
for MCP is disabled, per-tool settings won't take effect.

Configuration File Method

You can also configure auto-approve directly in the configuration file:

Auto-approve in Configuration 
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/safe/path"],
      "alwaysAllow": [
        "read_file",
        "list_directory",
        "get_file_info"
      ]
    },
    "database": {
      "command": "python",
      "args": ["-m", "mcp_server_postgres"],
      "alwaysAllow": [
        "query_read_only"
      ]
    }
  }
}

Security Considerations

  • Only auto-approve safe operations: Read-only operations are generally safe
  • Avoid auto-approving write operations: File writes, database updates, API calls that modify data
  • Review tool descriptions: Understand what each tool does before auto-approving
  • Use project-level config for team safety: Share safe auto-approve settings with your team

Writing Effective Tool Descriptions

Good tool descriptions help Bob understand when and how to use tools correctly.

Tool Name

  • Use descriptive, unambiguous names
  • Follow naming conventions (snake_case)
  • Avoid generic names like "process" or "handle"
  • Example: fetch_user_profile not get_data

Tool Description

  • Comprehensive summary of purpose
  • Explain use cases and scenarios
  • Mention prerequisites or requirements
  • Describe expected outcomes

Parameter Descriptions

  • Explain purpose of each parameter
  • Specify data type and format
  • Provide examples when helpful
  • Indicate if required or optional

Example: Good vs. Bad Descriptions

❌ Bad Tool Description 
{
  "name": "get_data",
  "description": "Gets data",
  "inputSchema": {
    "type": "object",
    "properties": {
      "id": {
        "type": "string",
        "description": "The ID"
      }
    }
  }
}
✅ Good Tool Description 
{
  "name": "fetch_user_profile",
  "description": "Retrieves a user's complete profile information from the database, including personal details, preferences, and account status. Use this when you need comprehensive user information for display or analysis. Requires a valid user ID. Returns a structured user profile object or an error if the user is not found.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "user_id": {
        "type": "string",
        "description": "Unique identifier for the user. Must be a valid UUID format (e.g., '123e4567-e89b-12d3-a456-426614174000'). This ID is typically obtained from authentication or user search operations."
      },
      "include_preferences": {
        "type": "boolean",
        "description": "Optional. If true, includes user preferences in the response. Default is false. Set to true when you need to display or modify user settings.",
        "default": false
      }
    },
    "required": ["user_id"]
  }
}

Using Custom Rules for Tool Preferences

You can guide Bob's tool usage through custom rules in your project or global settings.

Custom Rules Example 
When working with the database:
- Always use read_only_query for SELECT operations
- Use write_query only for INSERT, UPDATE, DELETE
- Prefer batch operations when processing multiple records
- Always validate input parameters before querying

When working with files:
- Use read_file for files under 1MB
- Use stream_file for larger files
- Always check file existence before reading
- Use relative paths when possible

Best Practices for Tool Usage

  • Be specific in requests: Clear requests help Bob choose the right tools
  • Review tool proposals: Check what Bob plans to do before approving
  • Use auto-approve wisely: Only for safe, frequently used operations
  • Monitor tool performance: Adjust timeouts if tools are slow
  • Provide feedback: Tell Bob if tool results aren't what you expected
  • Document custom tools: Write clear descriptions for tools you create
